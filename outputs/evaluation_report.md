## 最差 3 条 Case 分析

### case_20

- 总分：2.5 / 5
- 单项分：
  - 有用性：2
  - 准确性：2
  - 真实性/不瞎编造：4
  - 语气友好：3
- 用户问题：退货流程太复杂了搞半天都不知道怎么操作
- 自动回复：抱歉给您带来困扰。退货流程一般包括：1.在订单页面点击退货 2.选择退货原因 3.提交申请等待审核 4.按照退货地址寄回商品。如果您在某个环节遇到问题，可以截图发给客服，我们会协助您操作。
- 人工参考：抱歉让您感到困扰。请问您卡在哪个环节了？我一步步帮您操作。
- 人工分析：用户已经表达了操作困难和不满。自动回复又重复了一遍退货流程——这恰恰是用户说搞不懂的东西。人工回复会问具体卡在哪一步，然后针对性帮忙。自动回复属于'答非所问'。
- 系统识别问题：
  - 回复偏通用，未主动承接处理，用户仍需自行查询或操作。
  - 回复没有命中用户核心诉求，存在答非所问风险。
  - 人工参考中包含主动服务动作，自动回复覆盖不足。

- 评分理由：mock 模式根据人工标注趋势、主动服务信号、推责表达和事实风险规则评分。

## 与人工标注的一致性验证

- case_08：人工认为属于“正确但没用”，系统应体现为准确性/真实性较高、有用性较低。
- case_14：人工认为建议反馈处理较好，系统应给出较高总分。
- case_20：人工认为重复流程、答非所问，系统应给出较低准确性和有用性。

这类验证不是要求自动评估完全复刻人工回复，而是检查评分是否能复现人工指出的主要问题类型。

## 局限性与改进

- 缺少真实订单、物流、商品和优惠券系统数据时，真实性只能基于文本证据判断。
- mock 模式可复现、可离线，但语义理解有限，适合演示流水线和做基线。
- LLM 模式理解能力更强，但会受模型版本、prompt、温度和网络/API 状态影响。
- 人工参考回复是弱金标准，不代表唯一正确答案；后续应扩大样本并引入多名标注员一致性检验。
